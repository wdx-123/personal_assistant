# 图片管理技术文档

> 适用范围：`personal_assistant` 当前仓库中的图片管理模块  
> 文档目标：替代《1-图片管理-现阶段》和《图片管理-重构操作文档》，统一说明当前实现、接入方式、运行机制与后续扩展原则。  
> 文档基准：以当前代码为准，而不是以旧方案或历史草稿为准。

## 1. 结论先行

当前仓库中的图片管理已经不是“待重构状态”，而是已经落地为一套可运行的分层方案，核心能力包括：

- `local` / `qiniu` 双存储驱动。
- `POST /api/system/image/upload` 多文件上传。
- 文件扩展名校验 + 文件内容 MIME 校验。
- 基于 `SHA-256 + size` 的秒传去重。
- 单次上传数量限制、单用户存储配额、上传并发控制。
- Redis 驱动的上传限流（全局 + 用户双层）。
- 图片记录软删除，物理文件由定时任务异步清理。
- 用户头像、组织头像等业务场景复用图片表。
- 本地静态目录挂载与七牛 CDN URL 并存。

需要特别注意的现状：

- 删除接口不会立即删除物理文件，只会先软删除数据库记录。
- 孤儿物理文件由定时任务统一清理，清理周期由 `task.image_orphan_cleanup_cron` 控制。
- 本地驱动当前返回的是相对 URL：`static.prefix + "/" + key`，并未使用 `storage.local.base_url`。
- 图片模块当前不读取 `upload.size` / `upload.path`，真实生效的是 `static.*` 与 `storage.*`。
- 七牛驱动为了支持重试，会先把单文件读入内存再上传，容量评估时必须考虑这一点。

## 2. 模块定位与边界

图片模块的职责不是“简单存文件”，而是负责把上传、存储、元数据、业务引用和清理流程串起来。

分层职责如下：

| 层级 | 目录 | 职责 |
| --- | --- | --- |
| Controller | `internal/controller/system/imageCtrl.go` | 接参、鉴权上下文提取、调用 Service、组装响应 |
| Service | `internal/service/system/imageSvc.go` | 编排上传、秒传、配额、软删、清理逻辑 |
| Repository | `internal/repository/system/imageRepo.go` | 图片表的纯数据库操作 |
| Entity / DTO | `internal/model/entity/image.go`、`internal/model/dto/request/imageReq.go`、`internal/model/dto/response/imageResp.go` | 定义图片表结构与接口契约 |
| Router | `internal/router/system/imageRouter.go` | 注册图片上传、删除、列表路由 |
| Infra | `pkg/storage/*`、`internal/core/storage.go` | 存储驱动抽象、驱动注册、当前驱动选择、熔断与重试 |
| Task | `task/enter.go`、`pkg/imageops/orphan_cleanup.go` | 孤儿文件异步清理 |

边界原则：

- Controller 不写上传/删除业务逻辑。
- Service 不直接操作 `global.DB`。
- Repository 不碰文件系统和第三方存储 SDK。
- 物理文件删除不通过 HTTP 自调用，而是通过驱动层直接删除。

## 3. 当前整体架构

```text
客户端
  -> /api/system/image/upload|delete|list
  -> ImageCtrl
  -> ImageService
      -> ImageRepository (MySQL/GORM)
      -> pkg/storage 当前驱动 (local/qiniu)
      -> pkg/imageops (软删与孤儿清理)
  -> images 表

本地访问链路：
  浏览器 -> /images/<key> -> Gin StaticFS -> ./static/images/<key>

异步清理链路：
  Delete 软删记录
  -> 定时任务 ImageOrphanCleanupTask
  -> 查找无活跃引用的 key
  -> 驱动删除物理文件
  -> 硬删除已软删记录
```

## 4. 数据模型

图片实体为 `internal/model/entity/image.go`，表名为 `images`，并已加入自动迁移。

核心字段如下：

| 字段 | 含义 |
| --- | --- |
| `ID` | 图片记录主键 |
| `Name` | 原始文件名 |
| `Type` | 文件扩展名，如 `.png` |
| `Size` | 文件大小，单位字节 |
| `Driver` | 存储驱动名，当前支持 `local` / `qiniu` |
| `Key` | 驱动内部对象键，不包含域名前缀 |
| `URL` | 前端可直接使用的访问地址 |
| `Category` | 图片分类 |
| `UploaderID` | 上传者用户 ID |
| `OrgID` | 所属组织 ID，可空 |
| `FileHash` | 文件内容哈希，当前使用 `SHA-256` |
| `HashAlgo` | 哈希算法标识，当前为 `sha256` |
| `DeletedAt` | 软删除标记，由基类 `MODEL` 提供 |

分类枚举定义在 `internal/model/consts/image_category.go`：

| 值 | 含义 |
| --- | --- |
| `0` | 未使用 |
| `1` | 系统 |
| `2` | 轮播 |
| `3` | 封面 |
| `4` | 插图 |
| `5` | 广告 |
| `6` | Logo |
| `7` | 头像 |

## 5. 对外接口

### 5.1 上传接口

- 路径：`POST /api/system/image/upload`
- 认证：需要 JWT
- 限流：额外挂载上传限流中间件
- Content-Type：`multipart/form-data`

表单字段：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `files` | file[] | 是 | 支持多文件 |
| `category` | int | 否 | 图片分类 |
| `driver` | string | 否 | 指定驱动，`local` 或 `qiniu`；为空则走当前默认驱动 |

返回项为图片数组，每项包含：

- `id`
- `url`
- `name`
- `type`
- `size`
- `category`
- `category_label`

### 5.2 删除接口

- 路径：`DELETE /api/system/image/delete`
- 认证：需要 JWT
- 请求体：`{"ids":[1,2,3]}`

说明：

- 接口执行的是“软删除图片记录”。
- 该接口成功不代表物理文件已经删除。
- 如果多个图片记录因秒传复用了同一个 `key`，只有当没有活跃引用后，定时清理任务才会真正删除物理对象。

### 5.3 列表接口

- 路径：`GET /api/system/image/list`
- 认证：需要 JWT
- 查询参数：`page`、`page_size`、`category`

默认值：

- `page` 默认 `1`
- `page_size` 默认 `10`

### 5.4 静态访问接口

当使用本地驱动时，Gin 会挂载静态目录：

- URL 前缀：`/images`
- 本地目录：`./static/images`

因此本地驱动下返回的 URL 形如：

```text
/images/20260308/abcd1234efgh5678.png
```

当使用七牛驱动时，返回的 URL 为完整 CDN/域名地址：

```text
https://your-domain/<key>
```

## 6. 启动与初始化链路

图片模块的启动不是单点完成，而是嵌在系统初始化链路中：

1. `internal/core/config.go` 读取 `configs/configs.yaml` 与环境变量。
2. `flag/flagSql.go` 自动迁移 `images` 表。
3. `internal/core/storage.go` 注册 `local` 与 `qiniu` 驱动，并选择当前驱动。
4. `internal/core/ratelimit.go` 初始化上传限流器。
5. `internal/router/router.go` 挂载图片路由与本地静态目录。
6. `task/enter.go` 注册孤儿图片清理定时任务。

这意味着图片功能依赖以下运行时组件：

- MySQL：保存图片元数据与软删除状态。
- Redis：上传限流器依赖。
- 本地文件系统或七牛：保存物理文件。

## 7. 上传流程

当前上传逻辑位于 `internal/service/system/imageSvc.go`，实际流程如下：

1. Controller 绑定 `category`、`driver`，并从 JWT 中提取 `uploaderID`。
2. Service 解析驱动：
   - 若请求里显式给出 `driver` 且该驱动已注册，则使用指定驱动。
   - 否则回退到 `storage.current` 对应的当前驱动。
3. 检查单次上传数量是否超过 `static.max_uploads`。
4. 预估本次上传总大小，并根据 `static.user_quota_mb` 做用户配额检查。
5. 逐文件执行上传：
   - 获取上传并发信号量。
   - 建立单文件 30 秒超时上下文。
   - 校验扩展名是否在 `static.allowed_types` 中。
   - 读取前 512 字节做真实 MIME 校验，防止伪造扩展名。
   - 计算文件内容的 `SHA-256`。
   - 以 `file_hash + size` 查询是否已有相同文件。
6. 若命中秒传：
   - 不再重复上传物理文件。
   - 直接创建一条新的图片记录，复用原图片的 `Driver`、`Key`、`URL`。
7. 若未命中秒传：
   - 调用具体驱动上传。
   - 用驱动返回的 `StorageObject` 构建 `images` 记录并入库。

这套流程的意义是：

- 避免重复上传相同文件。
- 让“图片记录”和“物理对象”变成多对一关系。
- 为后续的引用计数式清理提供基础。

## 8. 删除与清理流程

### 8.1 删除接口的真实语义

当前删除实现已经不是旧文档中的“同步 HTTP 删除文件”，而是：

1. 对传入 `ids` 去重并过滤 `0` 值。
2. 执行 Repository 软删除。
3. 立即返回成功。

对应实现位于：

- `pkg/imageops/soft_delete.go`
- `internal/service/system/imageSvc.go`

### 8.2 为什么改成异步清理

这样设计主要解决两个问题：

- 秒传场景下，一个物理对象可能被多条图片记录共享，删除任意一条记录都不能直接删文件。
- 文件系统与对象存储删除不具备数据库事务能力，异步清理更容易做幂等和补偿。

### 8.3 孤儿文件清理流程

定时任务 `ImageOrphanCleanupTask` 会执行：

1. Repository 查询所有“已软删除且没有活跃引用”的 `key`。
2. 按记录中的 `driver` 找到对应驱动。
3. 调用驱动的 `Delete(ctx, key)` 删除物理对象。
4. 仅对物理删除成功的 `key`，执行数据库硬删除收尾。

关键特性：

- 若驱动不存在，任务会跳过并记录告警。
- 若物理删除失败，记录仍会保留，等待下次任务重试。
- 驱动删除要求幂等，资源不存在也应视为成功。

默认清理周期：

- `task.image_orphan_cleanup_cron`
- 默认值：`@daily`

## 9. 存储驱动设计

### 9.1 驱动抽象

统一接口位于 `pkg/storage/driver.go`：

- `Name()`
- `Upload(ctx, r, filename)`
- `Delete(ctx, key)`

驱动管理器位于 `pkg/storage/manager.go`，负责：

- 注册驱动
- 保存当前驱动
- 按名称解析驱动

### 9.2 本地驱动

本地驱动位于 `pkg/storage/local/local.go`，行为如下：

- 对象实际落盘在 `static.path`。
- 对象键格式为 `[key_prefix/]<yyyyMMdd>/<随机串><ext>`。
- URL 由 `static.prefix + "/" + key` 生成。
- 删除时直接 `os.Remove`。
- 上传时按块写入文件，并检查 `context.Context` 取消信号。

注意：

- 配置结构中虽然有 `storage.local.base_url`，但当前本地驱动没有使用它生成 URL。
- 因此本地驱动返回的是相对路径，不是完整域名。

### 9.3 七牛驱动

七牛驱动位于 `pkg/storage/qiniu/qiniu.go`，行为如下：

- 使用 `storage.qiniu.bucket`、`domain`、`access_key`、`secret_key`。
- 对象键格式与本地驱动一致，可带 `key_prefix`。
- 上传成功后返回完整 URL：`<domain>/<key>`。
- 删除时若七牛返回 `612`，视为资源不存在并按成功处理。

### 9.4 熔断与重试

在 `internal/core/storage.go` 中：

- 本地驱动直接注册。
- 七牛驱动会被熔断器包装。

当前策略：

- 连续失败 5 次后打开熔断。
- Open 状态持续 15 秒。
- Half-Open 最多允许 3 个探测请求。

七牛上传还带指数退避重试，最多重试 3 次。

### 9.5 一个重要的运维差异

本地驱动是流式写盘；七牛驱动为了支持失败重试，会先把文件读入内存。  
因此在以下场景要格外注意内存：

- `static.max_size` 较大
- `static.max_concurrent_uploads` 较大
- 上传流量集中
- 默认驱动为 `qiniu`

## 10. 配置说明

图片模块真正依赖的配置主要是下面这些：

### 10.1 存储驱动

```yaml
storage:
  current: "qiniu"
  local:
    base_url: ""
    key_prefix: ""
  qiniu:
    bucket: ""
    domain: ""
    key_prefix: ""
    access_key: ""
    secret_key: ""
```

含义：

- `storage.current`：当前默认驱动。
- `storage.local.key_prefix`：本地对象键前缀。
- `storage.qiniu.*`：七牛对象存储参数。

### 10.2 静态文件与上传限制

```yaml
static:
  path: "./static/images"
  prefix: "/images"
  max_size: 16
  max_uploads: 15
  allowed_types: [".png", ".jpeg", ".gif", ".jpg", ".bin"]
  max_concurrent_uploads: 50
  user_quota_mb: 50
```

含义：

- `path` / `prefix`：本地驱动与静态访问使用。
- `max_size`：单文件大小上限，单位 MB。
- `max_uploads`：单次请求允许上传的最大文件数。
- `allowed_types`：扩展名白名单，同时会推导 MIME 白名单。
- `max_concurrent_uploads`：服务内并发上传上限。
- `user_quota_mb`：单用户总存储空间配额。

### 10.3 上传限流

```yaml
rate_limit:
  upload:
    global_limit: 100
    global_window_sec: 60
    user_limit: 10
    user_window_sec: 60
```

说明：

- 全局限流保护整个上传接口。
- 用户级限流限制单用户频率。
- Redis 故障时中间件会降级放行，不会直接阻断业务。

### 10.4 定时清理

```yaml
task:
  image_orphan_cleanup_cron: "@daily"
```

说明：

- 控制孤儿图片物理清理的执行周期。
- 多实例部署时可配合任务分布式锁，避免重复清理。

### 10.5 容易混淆但当前未参与图片模块的配置

- `upload.size`
- `upload.path`
- `storage.local.base_url`

当前这些配置已经入模或已绑定环境变量，但不参与图片模块的实际上传与 URL 返回逻辑。

## 11. 业务接入方式

图片模块当前已经被业务层复用，最典型的是：

- 用户头像更新
- 组织头像更新

接入模式如下：

1. 业务先调用图片上传接口获得 `image.id` 与 `image.url`。
2. 业务表保存图片 ID 作为稳定引用。
3. 业务展示时可直接使用 `url`，也可保留业务字段中的头像 URL。
4. 当头像替换时：
   - 新图片会被标记为 `CatAvatar`
   - 旧图片记录会被软删除
   - 物理文件等待定时任务清理

这套模式的优点是：

- 业务表不会直接耦合底层存储驱动。
- 后续从本地切到云存储时，业务层几乎不用改。
- 可通过图片表统一做配额、分类、审计和清理。

对后续新增业务的建议：

- 优先保存图片 ID，不要只存裸 URL。
- 如果业务存在“替换图片”语义，建议在同一事务内更新业务记录并软删旧图。
- 如果图片是程序生成的，可复用 `ImageService.SaveGeneratedImage` 的思路统一入库。

## 12. 观测与排障

图片服务默认会纳入 service trace 模块，常见链路包括：

- `image.Upload`
- `image.Delete`
- `image.List`
- `image.CleanOrphanFiles`

排障时建议按下面顺序看：

1. Controller 日志，确认请求是否成功绑定。
2. Service 日志，确认是校验失败、配额不足、驱动异常还是数据库异常。
3. 若使用七牛，检查熔断器是否已打开。
4. 若删除后文件仍存在，先看是否只是软删未到清理周期。
5. 若本地 URL 可返回但文件访问 404，检查 `static.path`、`static.prefix` 与落盘路径是否一致。

常见失败原因：

| 现象 | 典型原因 |
| --- | --- |
| 上传返回“参数错误” | 表单字段不符合接口要求 |
| 上传返回“请选择要上传的文件” | `files` 字段缺失或为空 |
| 上传返回“不支持的文件类型” | 扩展名不在白名单 |
| 上传返回“文件内容类型不合法” | MIME 校验未通过 |
| 上传返回“存储空间不足” | 超过用户配额 |
| 上传频繁出现 429 | 全局或用户级限流触发 |
| 删除后图片还能访问 | 记录只是软删，尚未执行孤儿清理 |
| 七牛偶发失败后持续报错 | 熔断器进入 Open，需要等待恢复窗口 |

## 13. 对旧文档的修正结论

本次统一文档需要明确修正以下历史信息：

1. 旧文档中的“删除通过自调用 HTTP 接口完成”已经过时。
   当前实现是“软删除 + 定时清理”，不需要额外实现一个删除图片文件的 HTTP 接口。

2. 旧文档中的“图片管理还处于待重构阶段”已经过时。
   当前仓库已经具备 Entity、DTO、Repository、Service、Controller、Router、Storage Driver 的完整链路。

3. 旧文档中的旧路由信息已经过时。
   当前接口路径是 `/api/system/image/*`，不是早期草稿中的 `/image/upload`。

4. 旧文档中以 `Path` 为中心描述图片定位方式已经不准确。
   当前核心字段是 `Driver + Key + URL`。

5. 旧文档里将“删除旧头像”等场景理解为立即物理删除，也不再准确。
   当前用户/组织头像替换只会软删旧图片，真正清理依赖孤儿清理任务。

## 14. 后续扩展建议

如果后续继续演进图片模块，建议按以下方向推进：

- 新增图片详情查询、按上传者过滤、按驱动过滤、按组织过滤。
- 让 `storage.local.base_url` 真正参与本地 URL 构造，便于反向代理或 CDN 场景。
- 为后台管理补充图片引用关系查询，便于判断某图片是否仍被业务使用。
- 为清理任务补充更细的指标与告警。
- 视需要增加 OSS / COS 等更多驱动，但继续保持 `pkg/storage` 抽象不变。
