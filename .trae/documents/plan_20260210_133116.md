我将按照以下步骤完善用户模块的功能，以满足 API 设计文档的要求：

### 1. 补充 DTO (数据传输对象)
在 `internal/model/dto/request/userReq.go` 中添加缺失的请求结构体：
- **`UserListReq`**: 用于用户列表查询，包含 `Page`, `PageSize`, `OrgID`, `Keyword`。
- **`AssignUserRoleReq`**: 用于分配角色，包含 `UserID`, `OrgID`, `RoleIDs` (数组)。

### 2. 升级 Repository (数据层)
修改 `internal/repository` 接口与实现，支持更复杂的查询：
- **接口更新** (`interfaces/userRepository.go`): 升级 `GetUserList` 方法，支持 `UserListReq` 过滤参数。
- **实现更新** (`system/userRepo.go`): 实现带有 `OrgID` (联表查询 `user_org_roles`) 和 `Keyword` (模糊查询用户名/手机号) 的动态过滤逻辑。

### 3. 升级 Service (业务逻辑层)
- **`PermissionService`**: 添加 `ReplaceUserRolesInOrg` 方法，用于全量替换用户在某组织下的角色（先清理旧角色，再分配新角色，并同步 Casbin）。
- **`UserService`**:
    - 实现 `GetUserList`: 调用 Repo 获取用户列表，并批量填充用户的角色信息（避免 N+1 查询）。
    - 实现 `GetUserDetail`: 获取用户详情。
    - 实现 `GetUserRoles`: 获取用户在指定组织下的角色列表。
    - 实现 `AssignRole`: 调用 `PermissionService` 完成角色分配。

### 4. 升级 Controller (控制层)
在 `internal/controller/system/userCtrl.go` 中添加以下管理接口：
- `GetUserList`: 处理 `/system/user/list` 请求。
- `GetUserDetail`: 处理 `/system/user/{id}` 请求。
- `GetUserRoles`: 处理 `/system/user/{id}/roles` 请求。
- `AssignRole`: 处理 `/system/user/assign_role` 请求。

### 5. 注册路由
在 `internal/router/system/userRouter.go` 中注册上述新接口，确保 URL 路径与 API 文档一致。
